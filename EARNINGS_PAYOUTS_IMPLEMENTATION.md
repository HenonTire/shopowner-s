# Earnings & Payouts Implementation Notes

## Local Codebase Analysis

This workspace contains the Flutter shop manager client. A Django `payment` app is not present locally, so backend models, services, Celery tasks, serializers, webhooks, permissions, row locking, audit logging, and database transactions could not be inspected or modified here.

Existing reusable client patterns found and reused:

- Authenticated backend repositories use `ApiConfig.baseUrl`, `http.Client`, and `AuthSessionStore.token`.
- Riverpod providers expose repository instances and async data.
- Pages use Material 3 widgets, `AppThemes.poppins`, summary cards, `DataTable`, loading/error/empty states, and `RefreshIndicator`.
- Dashboard drawer navigation already contains payment-related entries, so those entries now open the new module instead of adding duplicate navigation architecture.

Existing payment-specific client code found:

- No earnings, wallet, payout, withdrawal, bank account, webhook, refund, or commission client models existed.
- Existing docs mention sales/orders and payment labels, but no earnings or payout endpoint contract existed.

## Earnings Flow Represented

The UI expects earnings to be generated by the backend after order/payment processing and returned as immutable history records with:

- date
- source order
- customer
- commission
- gross amount
- platform fee
- net earnings
- status

Supported statuses in the client are `pending`, `available`, `withdrawn`, `refunded`, and `cancelled`.

Refunds and cancelled orders are represented as statuses because local backend refund/cancellation logic was not available to inspect.

## Payout Flow Represented

The payout UI expects the backend to enforce the final production invariants:

- user owns the earnings and payout account
- available balance is sufficient
- minimum and maximum withdrawal rules
- duplicate pending/processing request prevention
- atomic balance updates
- idempotency
- audit logging

The client validates the same constraints before submission for fast feedback, but backend validation remains authoritative.

Supported payout statuses are `pending`, `approved`, `processing`, `paid`, `failed`, and `rejected`.
